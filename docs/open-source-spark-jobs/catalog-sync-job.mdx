A searchable data catalog is only useful if it stays in sync with what's actually in your warehouse. The **Data Catalog Sync Job** scans every catalog in your IOMETE environment and indexes the tables it finds. Once indexed, tables show up on the **Data Catalog** page, where you can add documentation, assign owners, define classifications, and set maintenance properties.

- **Version:** `5.0.0` — requires IOMETE `3.16.x` or later
- **Source:** [View on GitHub](https://github.com/iomete/iomete-marketplace-jobs/tree/main/iomete-catalog-sync)

## How It Works

Once the job is set up, it does its work on autopilot. It runs on a schedule (hourly by default), scans every catalog in your environment, and indexes the tables it finds.

You can change the schedule or trigger a run manually whenever you want. Anything matching an [exclusion rule](#exclusion-rules) is skipped during the run.

## Running the Job

Most of the time the schedule handles things for you. If you need a fresh index right now (say, after creating a batch of new tables), open the job and click <FlexButton label='Run' primary><Play size={12} weight="fill" /></FlexButton>.

## Deleted Tables

Dropping a table doesn't erase its history from the catalog, which is useful when you need to know what existed before and when it disappeared. When a table is dropped from a source catalog, the sync job marks it as **deleted** and hides it from the default view rather than removing the record.

Use the **Deleted** toggle in the Data Catalog UI to review what was removed and when.

<Img
  src="/img/spark-job/marketplace/catalog-sync/data-catalog-deleted.png"
  alt="Data Catalog deleted tables view"
/>

:::note
Deleted entries are kept indefinitely for now. Automatic cleanup of old deleted entries is planned for a future release.
:::

## Exclusion Rules

Not every catalog needs to be indexed. For example, you may want to skip:

- An external catalog (e.g., Oracle) with millions of tables that would clutter search results
- Catalogs or schemas containing sensitive data that shouldn't appear in the Data Catalog
- Personal or sandbox environments used for development

Exclusion rules let you skip specific catalogs, namespaces (schemas/databases), or tables during sync.

### Default Rule

If you want one rule to cover everything, start here. The default rule applies at **all levels**: catalogs, schemas, and tables. Any asset with a matching property is excluded from indexing.

```json
{
  "exclusion_rules": {
    "default_rule": {
      "filter_by_properties": {
        "iomete.governance.index": "false"
      }
    }
  }
}
```

With this rule in place, setting `iomete.governance.index=false` on any catalog, schema, or table excludes it from the sync. It's the simplest setup: one rule, applied wherever you need it via properties.

### Excluding Catalogs

When you want catalog-level control, you can exclude catalogs **by name**, **by property**, or both.

#### By Name

List catalog names directly. These catalogs are always skipped, regardless of their properties.

```json
{
  "exclusion_rules": {
    "catalogs": {
      "names": ["sandbox_dev", "legacy_warehouse", "user_personal_spaces"]
    }
  }
}
```

#### By Property

Set `iomete.governance.index` to `false` on the catalog in the **Admin Portal** when you create or edit it.

<Img
  src="/img/spark-job/marketplace/catalog-sync/exclude-catalog.png"
  alt="Setting iomete.governance.index property on a catalog in the Admin Portal"
/>

Then add the matching filter:

```json
{
  "exclusion_rules": {
    "catalogs": {
      "filter_by_properties": {
        "iomete.governance.index": "false"
      }
    }
  }
}
```

#### Combining Both

You can use names and properties together. A catalog is excluded if it matches **either** condition.

```json
{
  "exclusion_rules": {
    "catalogs": {
      "names": ["sandbox_dev", "legacy_warehouse"],
      "filter_by_properties": {
        "iomete.governance.index": "false"
      }
    }
  }
}
```

### Excluding Namespaces (Schemas/Databases)

Sometimes you want to keep a catalog in the index but skip a few schemas inside it. Namespace exclusion is property-based. Mark a namespace for exclusion by setting the property in SQL:

```sql
ALTER DATABASE my_catalog.my_schema SET DBPROPERTIES ('iomete.governance.index' = 'false');
```

Verify the property:

```sql
DESCRIBE DATABASE EXTENDED my_catalog.my_schema;
```

To re-enable indexing later, flip the property back:

```sql
ALTER DATABASE my_catalog.my_schema SET DBPROPERTIES ('iomete.governance.index' = 'true');
```

Then add the exclusion rule:

```json
{
  "exclusion_rules": {
    "schemas": {
      "filter_by_properties": {
        "iomete.governance.index": "false"
      }
    }
  }
}
```

### Excluding Tables

Tables work the same way as namespaces. Set a property on the table, then define the matching rule.

```sql
ALTER TABLE my_catalog.my_schema.my_table SET TBLPROPERTIES ('iomete.governance.index' = 'false');
```

Verify the property:

```sql
SHOW TBLPROPERTIES my_catalog.my_schema.my_table;
```

To remove the property later:

```sql
ALTER TABLE my_catalog.my_schema.my_table UNSET TBLPROPERTIES ('iomete.governance.index');
```

Then add the exclusion rule:

```json
{
  "exclusion_rules": {
    "tables": {
      "filter_by_properties": {
        "iomete.governance.index": "false"
      }
    }
  }
}
```

#### Multiple Properties (OR Logic)

When a table exclusion rule lists multiple properties, a table is excluded if **any one** of them matches.

```json
{
  "exclusion_rules": {
    "tables": {
      "filter_by_properties": {
        "iomete.governance.index": "false",
        "hidden": "true"
      }
    }
  }
}
```

Here, a table is excluded if `iomete.governance.index` is `false` **or** `hidden` is `true`. It doesn't need to match both.

### Full Example

A complete configuration that uses every exclusion level:

```json
{
  "exclusion_rules": {
    "catalogs": {
      "names": ["sandbox_dev", "legacy_warehouse", "user_personal_spaces"],
      "filter_by_properties": {
        "iomete.governance.index": "false"
      }
    },
    "schemas": {
      "filter_by_properties": {
        "iomete.governance.index": "false"
      }
    },
    "tables": {
      "filter_by_properties": {
        "iomete.governance.index": "false",
        "hidden": "true"
      }
    },
    "default_rule": {
      "filter_by_properties": {
        "iomete.governance.index": "false"
      }
    }
  }
}
```
